## Bug Fix 1 — `windowing_extensions_not_present`

**Symptom:** The app crashed at startup with:
```
[error] Failed to create Vulkan instance: windowing_extensions_not_present
```

**Root cause:** `VulkanContext::init()` was building the Vulkan instance without telling vk-bootstrap which windowing extensions GLFW needs. On Windows this silently worked because vk-bootstrap's defaults happen to cover `VK_KHR_win32_surface`. On Linux, it needs either `VK_KHR_xcb_surface` or `VK_KHR_wayland_surface` (depending on the display server) — neither of which was being requested.

**Fix:** Query GLFW for its required extensions before building the instance and pass them to `enable_extensions()`:

```cpp
uint32_t glfwExtCount = 0;
const char** glfwExts = glfwGetRequiredInstanceExtensions(&glfwExtCount);
// then: .enable_extensions(std::vector<const char*>(glfwExts, glfwExts + glfwExtCount))
```

**File:** `src/VulkanContext.cpp`

## Bug Fix 2 — vcpkg Vulkan loader shadowing the system loader

**Symptom:** After fix 1, `glfwGetRequiredInstanceExtensions()` returned `null`, meaning GLFW couldn't find Vulkan at all — even though `vulkaninfo` showed everything working fine.

**Root cause:** The binary's RPATH pointed to vcpkg's own `libvulkan.so.1` (inside the build directory). This is a stub loader that has no ICD (Installable Client Driver) configuration — it doesn't know where `radeon_icd.json` lives. The system loader at `/usr/lib/libvulkan.so.1` reads `/usr/share/vulkan/icd.d/` correctly, but the binary was never reaching it.

**Diagnosis:** `ldd ./build/linux-debug/vulkan-renderer` showed:
```
libvulkan.so.1 => /home/.../build/linux-debug/vcpkg_installed/x64-linux/debug/lib/libvulkan.so.1
```

**Fix:** In `CMakeLists.txt`, before `find_package(Vulkan REQUIRED)`, explicitly find and cache the system Vulkan library on Linux so the vcpkg version is overridden:

```cmake
if(UNIX AND NOT APPLE)
    find_library(_vulkan_system_lib NAMES vulkan
        PATHS /usr/lib /usr/local/lib /usr/lib/x86_64-linux-gnu
        NO_DEFAULT_PATH NO_CMAKE_FIND_ROOT_PATH)
    if(_vulkan_system_lib)
        set(Vulkan_LIBRARY "${_vulkan_system_lib}" CACHE FILEPATH "..." FORCE)
    endif()
endif()
```

Windows presets are unaffected — they use their own `base-windows` preset with no Linux-specific overrides.

**File:** `CMakeLists.txt`

## Bug Fix 3 — Semaphore reuse validation error

**Symptom:** Renderer started and triangle rendered, but the validation layers fired continuously:
```
VUID-vkQueueSubmit-pSignalSemaphores-00067
pSignalSemaphores[0] may still be in use by VkSwapchainKHR
```

**Root cause:** `renderFinishedSemaphores` were indexed by `m_currentFrame` (0 or 1), but the swapchain has 4 images. After 2 frames, the CPU recycled frame slot 0 and tried to signal `renderFinished[0]` again — but the presentation engine hadn't fully released it from the previous use of that slot. The fence only guarantees GPU *rendering* is complete; it doesn't guarantee the presentation engine has consumed the semaphore.

**Fix:** Index `renderFinishedSemaphores` by `imageIndex` (the acquired image index), not `m_currentFrame`. Allocate one per swapchain image (`swap.imageCount() = 4`).

The safety guarantee: the swapchain only re-issues image N after its previous presentation is complete. So by the time `vkAcquireNextImageKHR` gives back image N, `renderFinished[N]` is guaranteed to be free.

`imageAvailableSemaphores` stay indexed by `m_currentFrame` — the fence ensures those are safe to reuse.

**Files:** `src/Renderer.h`, `src/Renderer.cpp`

**Reference:** https://docs.vulkan.org/guide/latest/swapchain_semaphore_reuse.html

### Key things I understood better from documenting:

**Image layouts:** Images aren't just pixel buffers — they have an internal arrangement optimised for a specific use. `UNDEFINED` means discard, `COLOR_ATTACHMENT_OPTIMAL` is for writing, `PRESENT_SRC_KHR` is for the display engine. The barriers between these layouts are the "fence" for the GPU.

**Semaphore vs fence:** Semaphores are GPU→GPU signals (ordering between queue operations). Fences are GPU→CPU signals (letting the CPU know a frame slot is done). Using the wrong one for the wrong direction is a common Vulkan mistake.

**Staging buffers:** Device-local memory (VRAM) is fast for the GPU but not CPU-writable. The solution is a two-step upload: write to a host-visible staging buffer, then use `vkCmdCopyBuffer` to transfer GPU-side.

**Descriptor sets:** Like a table of pointers to resources (textures, UBOs). You define the slot types in a layout, allocate a set from a pool, then fill the slots via `vkWriteDescriptorSets`. Bind the set before a draw call.
