### Deep Dive Study — Vulkan M1 Internals

Built on yesterday's [[03-Learnings/Vulkan-Object-Chain]] overview with a comprehensive deep dive into the mechanics behind each object. Documented in [[03-Learnings/Vulkan-M1-Deep-Dive]].

Topics covered in depth:

- **The Vulkan loader dispatch chain** — how `libvulkan.so` discovers ICDs and layers, how `VK_INSTANCE_LAYERS` injects the validation layer into the call path, and why validation has a performance cost (middleware in every call).
- **Physical device selection internals** — the full list of queries vk-bootstrap runs during `select()`, and how the `pNext` feature chain mechanism works (`VkPhysicalDeviceFeatures2` -> `Vulkan13Features` -> `Vulkan12Features`).
- **Device vs instance extensions** — why `VK_KHR_swapchain` is a device extension but surface creation functions are instance extensions, and what this means for the creation order.
- **Queue submission mechanics** — the full `VkSubmitInfo2` structure, how double buffering overlaps CPU recording with GPU execution, and the timing diagram for `MAX_FRAMES_IN_FLIGHT = 2`.
- **Semaphores vs fences** — the exact rule (semaphores = GPU-to-GPU, fences = GPU-to-CPU), the complete signal/wait flow for one M1 frame, and the common mistake of trying to CPU-wait on a semaphore.
- **The exact two barriers for M1** — every field of both `VkImageMemoryBarrier2` structs explained with the specific `synchronization2` stage/access values and why each is set.
- **Graphics pipeline fixed-function state** — all 8 state blocks M1 needs, why viewport/scissor are dynamic (avoids pipeline recreation on resize), and why the pipeline layout is empty (no uniforms in M1).
- **Shader execution flow** — how `gl_VertexIndex` drives the hardcoded triangle, how the rasteriser interpolates `fragColor`, and why `VkShaderModule` is destroyed after pipeline compilation.
- **What validation layers actually check** — the 5 categories (parameter, lifetime, sync, threading, best practices) and the most common M1 errors to watch for.

## Code Decisions & Architecture Notes

- **Barrier values confirmed for M1:** `TOP_OF_PIPE/NONE -> COLOR_ATTACHMENT_OUTPUT/WRITE` before rendering, `COLOR_ATTACHMENT_OUTPUT/WRITE -> BOTTOM_OF_PIPE/NONE` after. These are the exact `synchronization2` masks that will go into `Renderer::drawFrame()`.
- **Dynamic viewport/scissor confirmed:** Pipeline will use `VK_DYNAMIC_STATE_VIEWPORT` and `VK_DYNAMIC_STATE_SCISSOR` to avoid recreation on window resize. Set per-frame via `vkCmdSetViewport`/`vkCmdSetScissor`.
- **Empty pipeline layout for M1:** No descriptor sets, no push constants. The triangle shader is fully self-contained. Layout becomes non-empty in M2 when the MVP UBO is added.
