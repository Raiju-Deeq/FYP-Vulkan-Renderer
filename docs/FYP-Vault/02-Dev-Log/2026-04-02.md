## What I Did

### Implemented VulkanContext.cpp — First Module Complete

Wrote the full implementation of `VulkanContext::init()` and `VulkanContext::destroy()`, building it up stage by stage with guidance. This is the foundational module that every other part of the renderer depends on.

**`init()` — 5 stages implemented:**

1. **Stage 1: VkInstance** — used `vkb::InstanceBuilder` to create the Vulkan instance with API version 1.3, validation layers (requested, not required — for DMU compatibility), and the default debug messenger.

2. **Stage 2: VkSurfaceKHR** — created the window surface via `glfwCreateWindowSurface()`. Learned this must happen before GPU selection so vk-bootstrap can verify presentation support.

3. **Stage 3: VkPhysicalDevice** — used `vkb::PhysicalDeviceSelector` with two feature structs:
   - `VkPhysicalDeviceVulkan13Features` — `dynamicRendering` + `synchronization2`
   - `VkPhysicalDeviceVulkan12Features` — `bufferDeviceAddress`
   - Learned these live in separate structs because they were promoted to core in different Vulkan versions (1.2 vs 1.3)

4. **Stage 4: VkDevice** — used `vkb::DeviceBuilder` which automatically propagates the required features from Stage 3. Also auto-enables `VK_KHR_swapchain` since a surface was provided.

5. **Stage 5: VkQueue** — retrieved the graphics queue handle and family index from the `vkb::Device`. Renderer will need both for command pool creation and command submission.

**`destroy()` — reverse teardown:**
- Device -> Surface -> Debug Messenger -> Instance
- Each guarded with `VK_NULL_HANDLE` check for safe partial cleanup
- Learned that `VkPhysicalDevice` and `VkQueue` are not owned handles and should not be destroyed

## Code Decisions & Architecture Notes

- **`request_validation_layers()` over `enable_validation_layers(true)`** — chosen for DMU lab compatibility. The labs may not have the Khronos validation layer installed, so a hard-fail would prevent the renderer from running at all. The `VK_INSTANCE_LAYERS` environment variable is the primary activation mechanism on my home machine.
- **C++20 designated initialisers for Vulkan structs** — `{ .sType = VK_STRUCTURE_TYPE_... }` zero-initialises all other fields automatically. This means every feature defaults to `VK_FALSE` and I only set the ones I need to `VK_TRUE`. Cleaner than memset or manually setting every field.
- **vkb wrapper locals kept alive across stages** — `vkbInstance`, `vkbPhysicalDevice`, and `vkbDevice` are stored as locals (not just their raw handles) because subsequent stages need the full vk-bootstrap wrapper. Discovered this during planning — the `PhysicalDeviceSelector` constructor takes a `vkb::Instance&`, not a `VkInstance`.
- **Doxygen `@details` on implementation** — added to both `init()` and `destroy()` since the header has `@brief` and `@param` but the implementation-specific logic (stage ordering, null-handle safety) belongs in the `.cpp`.
