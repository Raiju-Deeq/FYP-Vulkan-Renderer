Architecture

Raiju Renderer is organised as a small explicit Vulkan application. Each module owns a narrow set of Vulkan handles, exposes accessors for dependent systems, and is destroyed in reverse construction order.

    Renderer::init()

    render loop

    shutdown and destroy in reverse order

Ownership model

Who owns what

Root handles

VulkanContext owns VkInstance, VkDevice, VkSurfaceKHR, debug messenger, and VMA allocator. It does not own VkPhysicalDevice or VkQueue.

Presentation resources

SwapChain owns VkSwapchainKHR, colour image views, depth images, depth allocations, and depth views. Swapchain images are driver-owned.

Frame resources

Renderer owns the command pool, command buffers, image-available semaphores, per-image render-finished semaphores, in-flight fences, and ImGui descriptor pool.

Draw resources

Pipeline, Mesh, and Material own the GPU objects needed to bind state and issue one indexed textured draw.

Design decisions

Why the code is shaped this way

Dynamic Rendering

The render path avoids legacy VkRenderPass and VkFramebuffer. Pipelines declare attachment formats, and frames bind actual views with vkCmdBeginRendering.

Four mesh pipelines

Wireframe and culling are fixed rasterizer state, so the application builds solid, solid culled, wireframe, and wireframe culled pipeline variants.

Per-image present semaphore

renderFinished semaphores are indexed by swapchain image, not frame slot, to avoid reusing a semaphore still held by presentation.

Staging uploads

Meshes and textures are copied through host-visible staging resources into device-local VMA allocations for correct GPU memory usage.
