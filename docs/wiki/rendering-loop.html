Frame guide

Rendering Loop

The renderer uses two CPU frame slots and a swapchain image ring. Each frame waits for a reusable CPU slot, acquires a presentable image, records Dynamic Rendering commands, submits them to the graphics queue, then presents the image.

Frame sequence

One call to Renderer::drawFrame

Wait for the frame slot fence

The CPU waits for m_inFlightFences[m_currentFrame] so it does not overwrite command buffers or semaphores still used by the GPU.

Acquire a swapchain image

SwapChain::acquireNextImage returns an image index and arranges for imageAvailable[currentFrame] to be signalled when the image is safe to write.

Reset fence and command buffer

The fence is reset only after successful acquire, preventing a deadlock if acquire fails before any GPU work is submitted.

Transition colour and depth images

vkCmdPipelineBarrier2 transitions the swapchain image to COLOR_ATTACHMENT_OPTIMAL and the depth image to DEPTH_ATTACHMENT_OPTIMAL.

Begin Dynamic Rendering

vkCmdBeginRendering receives the acquired colour image view and matching depth image view. No render pass or framebuffer is created.

Bind and draw

The active mesh pipeline is selected from debug UI state, the material descriptor set is bound, push constants pass MVP and debug mode, then the mesh records vkCmdDrawIndexed.

Render ImGui

ImGui draw data is recorded inside the same Dynamic Rendering scope so the debug overlay appears over the renderer output.

Transition to present

The colour image moves from COLOR_ATTACHMENT_OPTIMAL to PRESENT_SRC_KHR so the presentation engine can display it.

Submit and present

Queue submit waits on imageAvailable, signals renderFinished[imageIndex], and signals the frame fence. Present waits on the per-image render-finished semaphore.

Synchronisation

Why the semaphore arrays are indexed differently

imageAvailable

Indexed by frame slot. The frame fence ensures a slot is not reused until its previous GPU submission has finished.

renderFinished

Indexed by swapchain image index. Presentation can hold this semaphore after queue submit, so tying it to the image prevents unsafe reuse.

inFlightFences

Indexed by frame slot. These are CPU-visible wait objects used to stop the CPU from getting too far ahead of GPU work.

MAX_FRAMES_IN_FLIGHT

Set to 2. The CPU can prepare one frame while the GPU executes another without sharing command buffers between active frames.

Resize recovery

What happens when the swapchain is out of date

drawFrame returns false on VK_ERROR_OUT_OF_DATE_KHR or VK_SUBOPTIMAL_KHR. Application::run then waits for the device to idle, waits for a non-zero framebuffer size, and rebuilds the dependent objects. 

renderer.waitIdle(ctx);
swap.rebuild(ctx, framebufferWidth, framebufferHeight);
renderer.recreateSwapchainSync(ctx, swap);
material.destroy(ctx);
destroyMeshPipelines(ctx, pipelines);
initMeshPipelines(ctx, swap.format(), swap.depthFormat(), pipelines);
material.init(ctx, loadedTexture, pipelines.solid.descriptorSetLayout());
renderer.recreateImGui(ctx, swap, window);
Implementation overview

Where this logic lives

Renderer

Frame resources, ImGui setup, draw loop, waiting, and teardown are handled by the renderer implementation in the frame data module.

Frame data

Command recording, image layout transitions, Dynamic Rendering setup, submit, and present are implemented in the frame data source files.

SwapChain

Image acquisition, image views, depth resources, and rebuild behaviour are owned by the swapchain module.

Pipeline

Dynamic Rendering pipeline formats and fixed-function state are constructed in the pipeline module used by the renderer.
