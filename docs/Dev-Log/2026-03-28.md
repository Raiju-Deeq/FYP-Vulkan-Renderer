## Code Decisions & Architecture Notes

- **vk-bootstrap for initialisation** — deliberate, documented decision. Instance creation, physical device selection, logical device setup, and swapchain construction are boilerplate that every professional Vulkan codebase abstracts. Writing them from scratch would consume a disproportionate share of development time for minimal additional learning. The underlying sequence is fully documented in Report §2 and the trade-off is analysed in §7.

- **Dynamic Rendering throughout** — `VK_KHR_dynamic_rendering` (core in Vulkan 1.3). No `VkRenderPass`. No `VkFramebuffer`. Ever. This is a structural decision, not a shortcut — it reduces complexity, reflects current industry direction, and produces a codebase that is easier to extend toward M6 and beyond.

- **M6 Gaussian splatting architecture sketch** — the splat pass sits in its own `GaussianSplat` module and composes onto the main scene via a second `vkCmdBeginRendering` call with `VK_ATTACHMENT_LOAD_OP_LOAD`, the same pattern used for the ImGui overlay. This keeps the core renderer clean and makes M6 genuinely additive rather than invasive.
